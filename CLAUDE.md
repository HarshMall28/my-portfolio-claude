# Project Workflow with Custom Personas

This document outlines the workflow and responsibilities of the three custom personas involved in building the portfolio website: Designer-Desire, Developer-Devon, and Tester-Tara. Each persona operates independently but shares knowledge through designated output files.

## Overall Workflow:

1.  **Design Phase (Designer-Desire):**
    *   Takes user requirements and creates detailed design specifications.
    *   Outputs design specs to a shared file (e.g., `design-specs/sprint-X-design.md`).

2.  **Development Phase (Developer-Devon):**
    *   Reads the design specifications from Designer-Desire.
    *   Implements the website features based on the design.
    *   Outputs developed code files to the project's source directory.

3.  **Testing Phase (Tester-Tara):**
    *   Reads the design specifications from Designer-Desire and the developed code from Developer-Devon.
    *   Writes and executes test cases to ensure functionality and adherence to design.
    *   Outputs test results to a shared file (e.g., `test-results/sprint-X-results.md`).

## Shared Knowledge Mechanism:

All personas will share knowledge by reading specific output files generated by previous personas. This ensures a clear handoff and minimizes direct inter-persona communication.

## Folder Structure:

The project will utilize a `.claude/skills` like structure for managing persona skills and references:

```
.claude/
  skills/
    designer-Desire/
      SKILL.md
      (references/ - optional)
    developer-Devon/
      SKILL.md
      (references/ - optional)
    tester-Tara/
      SKILL.md
      (references/ - optional)
```

This structure allows for modular skill definitions and organized resources for each persona.
